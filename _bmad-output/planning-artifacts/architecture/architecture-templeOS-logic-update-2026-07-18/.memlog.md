- (constraint) Parent architecture is binding: clean DB reset, super-admins are platform-wide, tenant-admin is a tenant-scoped role through tenant_memberships, persons are global, roles are tenant-scoped, tenant login resolves from *.trytempleos.com, and privileged actions use one audit_log.
- (constraint) Current checkout logic still uses admin_users, getPilotTenant, adminId sessions, requireSuperAdmin tenant-local checks, created_by/recorded_by legacy admin references, and seed scripts that attach data to the first tenant.
- (decision) Adopt a two-session model: super-admin session and tenant session are separate modules, cookie names, payloads, and route guards. Tenant session is created only after hostname resolves tenant and phone resolves person plus active membership.
- (decision) All existing tenant dashboard data routes keep tenantId-from-session scoping but replace adminId and AdminUser assumptions with personId, membershipId, and role/capability checks.
- (decision) Replace admin management logic with tenant member management: tenant-admins can create/reuse persons by phone, create tenant_memberships, and assign allowed roles inside their tenant only; super-admins can do the same across tenants through provisioning/control-plane routes.
- (decision) New authored tenant records should reference tenant_membership_id for contextual actions, not person_id alone; this preserves which temple-role context performed the action.
- (decision) Tenant session cookies carry identity pointers and role hints, but authorization guards must re-read active membership, role assignment, and role-definition capability state before privileged tenant access; role removal/deactivation must take effect without waiting for cookie expiry.
- (decision) Tenant guarded requests must resolve the current request host through one trusted-host helper and assert the resolved tenant matches the tenant session; tenant cookies are host-only in production and local host overrides are development-only.
- (decision) Both provisioning and tenant-local member routes must use a shared membership mutation service for phone normalization, person reuse, membership create/reactivate, role assignment, and audit behavior; route handlers must not orchestrate these multi-table writes directly.
- (decision) Audit actor model uses actor_type values super_admin, tenant_membership, system, and provider; actor_id is nullable only for system/provider actors; tenant_id is nullable only for global platform actions like role catalog changes.
- (decision) Tenant login does not create unaffiliated persons on failed membership checks; it finds a person by normalized phone first, and person creation happens only through provisioning, member management, super-admin bootstrap, or a successful membership-bound flow.
- (decision) Correction: tenant login does not create persons; person creation for tenant access happens through provisioning, tenant member management, or super-admin bootstrap before login can succeed.

- (decision) Clarify AD-7: tenant-admins manage members only inside their tenant, while super-admins may manage any tenant's members and roles only through super-admin routes with explicit tenant_id and audit logging.
