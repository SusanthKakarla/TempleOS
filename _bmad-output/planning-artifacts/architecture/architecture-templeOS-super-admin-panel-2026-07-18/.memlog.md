- (direction) User accepted Fast path and requested a plan for the Super Admin panel; deliver an architecture spine plus a short implementation plan for the builders.
- (constraint) Existing MVP and PRD are one-pilot validation artifacts: Super Admin dashboard, self-serve temple onboarding, tenant approval, billing, and WhatsApp self-serve connection UI were explicitly non-goals for the original MVP.
- (constraint) Current database is tenant-aware: tenant-owned tables carry tenant_id, admin_users belongs to one tenant, whatsapp_accounts belongs to one tenant, and webhooks resolve tenant from Meta phone_number_id.
- (constraint) Current provisioning is pilot-only: migration 002 inserts first Pilot Temple, getPilotTenant selects oldest tenant, and seed, seed:admin, seed:whatsapp all attach changes to that tenant.
- (decision) Adopt a layered operator-control-plane extension to the existing Next app: UI routes call server route handlers, route handlers call lib/db repositories, repositories own SQL mutations. Binds operator panel, tenant provisioning, admin provisioning, WhatsApp account linking. Prevents UI components or scripts from each inventing tenant creation SQL and session rules.
- (decision) Separate platform operators from tenant admins in the authorization model. Platform operators may create and manage temples across tenants; tenant-local super_admin remains limited to admins inside its own tenant. Prevents tenant super_admin from becoming accidental cross-tenant root access.
- (decision) Create one canonical provisioning service that atomically creates a tenant, first tenant super_admin, and optional WhatsApp account link in one transaction. Both CLI and operator UI must call this service rather than duplicating provisioning steps. Prevents partial tenant setup and divergent CLI-vs-UI behavior.
- (decision) Operator panel is not self-serve temple onboarding. There is no public signup, approval queue, tenant switching for tenant admins, billing, subscription management, or temple-owned WhatsApp connection wizard in this slice. Prevents scope creep from turning provisioning into a marketplace/onboarding product.
- (decision) Tenant identity is immutable for tenant-owned data after provisioning: dashboards and APIs derive tenant_id from session or WhatsApp phone_number_id, never from client-supplied tenant selectors. Operator APIs may accept tenant IDs only inside operator-authorized routes. Prevents cross-tenant leakage and client-side tenant spoofing.
- (decision) Replace pilot-only seed/admin/WhatsApp paths with explicit tenant-targeted provisioning commands; keep getPilotTenant only for backwards-compatible local demo bootstrap or retire it when the new CLI is adopted. Prevents new temples from silently attaching to the first tenant.
- (decision) Provisioning writes must be audit-friendly: every operator-created or operator-updated tenant, admin, or WhatsApp account action should record actor, target tenant, action, timestamp, and metadata once audit tables exist; until then, server logs must include the same action boundary. Prevents untraceable cross-tenant administrative changes.
- (decision) WhatsApp setup remains operator-managed: the panel stores and updates Meta phone_number_id and business account ID mappings, but does not implement OAuth, embedded signup, webhook auto-registration, or template approval workflows. Prevents Meta onboarding complexity from blocking the provisioning panel.
- (decision) Operator UI starts with create/list/view/update status, not destructive deletion. Tenant deletion, data export, tenant transfer, and tenant impersonation are deferred until explicit operational rules exist. Prevents accidental production data loss or ambiguous ownership changes.
- (assumption) Initial platform operators can be allowlisted by phone number using a separate platform_operators table or env-seeded bootstrap, reusing Firebase phone OTP verification; exact table name is an implementation detail but the operator identity store must not live in tenant admin_users.
- (version) Project stack reality checked from package.json: Next.js 16.2.10, React 19.2.4, TypeScript, pg 8.22.0, Firebase 12.16.0, firebase-admin 14.2.0, zod 4.4.3, Vitest 4.1.10; deployment target remains Railway app plus Railway Postgres from MVP docs.
- (decision) Operator authorization must use session helpers, cookie names, and payload types distinct from tenant dashboard sessions; tenant session cookies must never authorize operator routes.
